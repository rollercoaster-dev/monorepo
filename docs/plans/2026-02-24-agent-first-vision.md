# Agent-First Development: A Vision for native-rd

**Date:** 2026-02-24
**Status:** Draft
**Owner:** Joe

## Context

This repo has been agent-built from the start. 92 commits, 247 source files, 253 tests — all generated by Claude Code with a solo human steering. The question isn't "should we use agents?" — it's "what infrastructure would let agents do 10x more, with 10x less human attention?"

This document maps the capability layers needed to reach that goal, inspired by [OpenAI's Codex team experience](https://openai.com/index/building-an-agent-that-builds-agents/) of shipping a million-line product with zero manually-written code. Their key insight: **when agents write all the code, the engineer's job becomes designing environments, specifying intent, and building feedback loops.** The scarce resource is human time and attention — everything else should be automated.

**Our constraint is sharper**: one human, not seven. That means every hour spent on scaffolding must compound harder. The system must be self-reinforcing — agents that improve the environment that makes agents more effective.

## The Five Capability Layers

### Layer 1: Repository Legibility

**Goal**: Make the repo self-documenting for agents. An agent should be able to navigate from a high-level task to the exact code, docs, and context it needs without human guidance.

**What to build:**

- **Root `AGENTS.md`** — A ~100-line table of contents, not an encyclopedia. Points to deeper sources of truth. This is the first thing every agent reads; it must be short, stable, and accurate.
- **In-repo `CLAUDE.md`** — Move project-specific instructions from global config into the repository. Version it. Keep it under 200 lines.
- **Structured `docs/` as system of record** — Index files for each directory (`docs/design/index.md`, `docs/architecture/index.md`). Cross-links between related documents. Freshness metadata (last-verified date) on each doc.
- **Progressive disclosure** — Agents start with `AGENTS.md` (the map), drill into `docs/` (the territory), then into source code. No document assumes you've read another — each links to its prerequisites.
- **Design docs as first-class artifacts** — Active plans, completed plans, and ADRs all versioned and co-located in `docs/`. When a plan is executed, its status updates in-repo. No plans live only in GitHub issues or human memory.

**Why this is Layer 1**: Everything downstream depends on agents understanding the codebase. Without repository legibility, more autonomy just means more confident mistakes.

---

### Layer 2: Code Enforcement

**Goal**: Encode architectural taste into tooling so agents can't drift, even across hundreds of PRs.

**What to build:**

- **Architectural boundary lints** — Custom ESLint rules that enforce dependency directions. Components don't import from screens. Screens don't import from navigation internals. Theme tokens are used instead of raw colors (we already caught this in #112).
- **Structural tests** — Jest tests that validate the shape of the codebase: every component has an `index.ts`, every screen has a styles file, every new component has a test file.
- **Naming convention enforcement** — Files, exports, theme tokens, and test descriptions follow predictable patterns. Agents can find things by convention, not by searching.
- **Agent-readable error messages** — When a lint rule fails, the error message includes remediation instructions written for an agent, not a human. Example: `"Import from 'src/components/Button' violates layer boundary. Screens may only import from component index files. See docs/architecture/layering.md for the dependency graph."`
- **File size limits** — Flag files over 300 lines. Agents produce better output when working with focused, single-responsibility modules.

**Why this is Layer 2**: Enforcement is what allows velocity without decay. The Codex team's insight: "once encoded, rules apply everywhere at once." For a solo dev, this is even more critical — you can't manually review everything.

---

### Layer 3: App Legibility

**Goal**: Agents can see, drive, and validate the running app. An agent that can't observe the UI is flying blind on a visual product.

**What to build:**

- **Maestro E2E test suite** — Agents author and run Maestro flows. Each user story from `docs/vision/user-stories.md` has a corresponding Maestro test. Agents validate their own UI changes by running the flow.
- **Simulator screenshot capture** — An MCP tool or CLI skill that takes a screenshot of the iOS simulator and returns it to the agent. Agents can visually verify layout, theme rendering, and empty states.
- **Agent-readable logs** — Metro bundler output, React Native console logs, and crash reports piped to a file the agent can read. Structured logging where possible (JSON lines).
- **Per-worktree app boot** — Each git worktree can boot its own Expo dev client instance. Agents working in isolation don't interfere with each other or with the human's running app.
- **Accessibility audit automation** — Agents can run the accessibility test suite (`src/__tests__/accessibility/`) against the running app and get structured pass/fail results.

**Why this is Layer 3**: This is the biggest unlock the Codex team described. Agents that can observe the app can validate their own work, reproduce bugs, and close the loop without human QA. For a neurodivergent-first app where visual themes and accessibility are core features, this is non-negotiable.

---

### Layer 4: Feedback Loops

**Goal**: Agent-to-agent quality gates that catch issues before a human ever looks at a PR.

**What to build:**

- **Self-review skill** — Before requesting external review, the agent reviews its own PR: runs tests, checks lint, verifies the diff against the issue requirements, and confirms no unrelated changes snuck in. This is a gate, not a suggestion.
- **Multi-agent review pipeline** — CodeRabbit (already in place) + Claude code review (already in place) + a dedicated test coverage analyzer. All run in parallel on every PR.
- **Quality scoring per domain** — Each area of the codebase (components, screens, navigation, themes, tests) gets a quality grade. Grades are stored in `docs/quality/` and updated by agents. When a grade drops, it becomes a task.
- **Review-to-task pipeline** — Review comments that aren't resolved in the PR become tracked issues. No feedback disappears into a closed PR thread.
- **Acceptance criteria validation** — Agents check their PR against the original issue's acceptance criteria before marking it ready for review. If criteria aren't met, the agent iterates, not the human.

**Why this is Layer 4**: The Codex team's "Ralph Wiggum Loop" — agents iterating with agent reviewers until all reviewers are satisfied. This is what removes the human bottleneck from the review process. The human reviews by exception, not by default.

---

### Layer 5: Entropy Management

**Goal**: Continuous garbage collection that prevents drift, tech debt accumulation, and "AI slop."

**What to build:**

- **Golden principles as lint rules** — Not prose in a doc, but enforced rules. "Use shared components, not hand-rolled helpers." "Validate at boundaries." "Theme tokens, not raw colors." Each principle that matters enough to write down matters enough to enforce mechanically.
- **Recurring cleanup agents** — Scheduled tasks (or skills triggered periodically) that scan for: stale documentation, dead code, pattern drift (e.g., three components doing the same thing differently), unused exports, test coverage gaps.
- **Doc gardening** — An agent that compares documentation claims against actual code behavior and opens fix-up PRs when they diverge. `docs/vision/product-vision.md` says "UI library: Tamagui or Gluestack" but we chose Unistyles — a gardening agent catches this.
- **Versioned tech debt tracker** — A `docs/quality/tech-debt.md` file that lists known debt, its severity, and its status. Agents consult this before making changes in affected areas.
- **Quality grade dashboard** — A single file (`docs/quality/grades.md`) with current quality grades across all domains. Agents read this to prioritize cleanup work. Humans read it to see where attention is needed.

**Why this is Layer 5**: The Codex team spent 20% of their week on manual cleanup before automating it. As a solo dev, you can't afford that. Automated entropy management is what makes high velocity sustainable.

## What Changes for the Human

When this system is working, the human's job becomes three activities:

1. **Specifying intent** — Writing prompts, acceptance criteria, user stories. Describing what the product should do, not how.
2. **Designing environments** — Building skills, hooks, linters, MCP tools that make agents effective. Investing in scaffolding that compounds.
3. **Taste decisions** — Reviewing agent output for product sense, UX judgment, and architectural coherence. The human is the arbiter of "does this feel right?" — everything else is agent work.

Everything else — implementation, testing, review, documentation, cleanup — is delegated to agents.

## Phased Rollout

Each phase builds on the previous. Phase 1 is prerequisite for everything.

### Phase 1: Repository Legibility (Immediate)

- Create root `AGENTS.md`
- Move project instructions into in-repo `CLAUDE.md`
- Add index files to each `docs/` subdirectory
- Add freshness metadata to existing docs
- Fix stale references (product-vision.md still references Tamagui/Gluestack)

### Phase 2: Code Enforcement (Short-term)

- Custom ESLint rules for architectural boundaries
- Structural tests for codebase shape
- Naming convention enforcement
- Agent-readable lint error messages

### Phase 3: App Legibility (Medium-term)

- Maestro E2E test setup and initial flow coverage
- Simulator screenshot MCP tool
- Structured logging for agent consumption
- Per-worktree Expo dev client boot

### Phase 4: Feedback Loops (Medium-term)

- Self-review skill (pre-review gate)
- Quality scoring system per domain
- Review-to-task pipeline
- Acceptance criteria validation in PR workflow

### Phase 5: Entropy Management (Ongoing)

- Promote golden principles from prose to lint rules
- Doc gardening agent (recurring)
- Tech debt tracker and quality grade dashboard
- Pattern drift detection
